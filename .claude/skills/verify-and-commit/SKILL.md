# 검증 + 커밋 (verify-and-commit)

변경 사항을 **안전하게 저장**하는 표준 절차. 핵심 원칙: **깨진 코드는 절대 커밋하지 않는다.**

## 절차

1. **변경 파악** — `git status --short`로 무엇이 바뀌었는지 확인한다.

2. **타입체크 (필수)** — `npx tsc --noEmit` 실행.
   - 에러가 있으면 **여기서 중단.** 커밋하지 말고 에러를 그대로 보고한다.

3. **빌드 (코드 변경이 클 때)** — `npm run build`.
   - 실패하면 중단하고 원인을 보고한다.

4. **런타임 검증 (브라우저에 보이는 변경일 때)** — 프리뷰 서버가 떠 있으면
   preview 도구로 리로드 → 콘솔 에러 확인 → 핵심 화면 스냅샷/스크린샷으로 실제 동작을 확인한다.
   문제가 있으면 소스를 고치고 2번부터 재검증한다.
   - 프리뷰로 확인 불가한 변경(빌드 도구·타입 선언·순수 문서 등)은 이 단계를 건너뛴다.

5. **커밋 (위 검증을 통과했을 때만)**
   - 관련된 파일끼리 **의미 단위로 묶어** 스테이징한다(`git add <경로>`). 무관한 변경을 한 커밋에 섞지 않는다.
   - 커밋 메시지: 명확한 한국어 요약 + 필요 시 본문. 끝에 트레일러
     `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.
   - `.claude/settings.local.json` 등 git이 무시하는 개인 파일은 건드리지 않는다.

6. **보고** — 만든 커밋을 `git log --oneline -N`으로 보여주고, 초보 사용자가 이해하도록
   "세이브 지점" 개념으로 쉽게 설명한다. **push는 명시적으로 요청받지 않으면 하지 않는다.**

7. **SIM 점검** — 응답 끝에 SIM 관점의 냉정한 오류 점검을 붙인다(프로젝트 규칙, `.claude/agents/sim.md`).

## 가드레일
- 검증 없이 "저장 완료/성공"이라고 말하지 않는다.
- 타입체크·빌드·런타임 중 하나라도 실패하면 **커밋 대신 문제를 보고**한다.
- 한 번에 너무 많은 것을 섞지 말고, 논리적으로 나눠 여러 커밋을 만든다.
